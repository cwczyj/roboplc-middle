# 消息传递机制

> RoboPLC 使用 Hub 消息总线实现 Worker 之间的消息传递。本文档详细说明消息类型、传递流程和设计决策。

## 🔄 消息类型

### DeviceControl
设备控制请求消息，用于控制 Modbus 设备。

**字段：**
```rust
pub struct DeviceControl {
    pub device_id: String,          // 目标设备 ID
    pub operation: Operation,         // 操作类型（SetRegister, GetRegister 等）
    pub params: JsonValue,           // 操作参数
    pub correlation_id: u64,        // 请求 ID，用于匹配响应
}
```

**操作类型：**
- `SetRegister`：设置寄存器
- `GetRegister`：读取寄存器
- `ReadBatch`：批量读取
- `WriteBatch`：批量写入
- `MoveTo`：移动机械臂
- `GetStatus`：查询状态

### DeviceResponse
设备控制响应消息，包含操作结果。

**字段：**
```rust
pub struct DeviceResponse {
    pub device_id: String,          // 目标设备 ID
    pub success: bool,             // 操作是否成功
    pub data: JsonValue,           // 操作结果数据
    pub error: Option<String>,    // 错误信息（如果有）
    pub correlation_id: u64,        // 请求 ID，匹配原始请求
}
```

### ConfigUpdate
配置更新通知消息，由 ConfigLoader 发送。

**字段：**
```rust
pub struct ConfigUpdate {
    pub config: String,  // JSON 格式的配置内容
}
```

### DeviceHeartbeat
设备心跳消息，保持连接活跃状态。

**字段：**
```rust
pub struct DeviceHeartbeat {
    pub device_id: String,       // 设备 ID
    pub timestamp_ms: u64,  // 时间戳（毫秒）
    pub latency_us: u64,     // 延迟（毫秒）
}
```

### SystemStatus
系统状态查询和响应消息。

**字段：**
```rust
pub struct SystemStatus {
    pub requester: String,      // 请求者
    pub respond_to: Sender<SystemStatusResponse>,  // 响应通道
    pub devices_count: u32,  // 设备总数
    pub system_healthy: bool,   // 系统健康状态
    pub uptime_secs: u64,      // 运行时间
}
```

## 📡 消息传递流程

### 1. RPC Worker → Hub

```
RPC Worker (JSON-RPC Client)
    │
    ├── 接收 JSON-RPC 请求
    │
    ├── 解析请求并生成 DeviceControl
    │
    ├── 生成唯一 correlation_id
    │
    └── 通过 Hub 发送 DeviceControl 消息
    │
    └── 等待 DeviceResponse
    │
    └── 将结果转换为 JSON-RPC 响应返回客户端
```

**关键点：**
- **通道机制**：RpcWorker 使用内部通道在 RpcHandler 和主循环间传递消息
- **响应匹配**：存储 pending_requests HashMap，通过 correlation_id 匹配响应
- **超时处理**：设置合理的超时时间

### 2. Hub 路由

```
Hub
    │
    ├── 接收来自 RPC Worker 的 DeviceControl
    │
    ├── DeviceManager 接收并路由
    │   │
    │   ├── 使用 worker_map 查找目标 ModbusWorker
    │   │   │
    │   └── 通过 Hub 发送 DeviceControl 到目标 Worker
    │
    │
    └── 等待 DeviceResponse
    │
    │
    │   └── ModbusWorker 发送 DeviceResponse
    │
    │
    │   └── DeviceManager 接收并匹配响应
    │
    │
    │   ├── 通过 correlation_id 查找原始请求
    │   │   ├── 如果找到，通过 response_tx 发送响应数据
    │   │   └── 移除 pending_requests 中的条目
    │
    │
    └── 如果未找到，记录警告日志
    │
```

**关键点：**
- **worker_map**：DeviceManager 维护设备 ID → Worker 名称的映射
- **pending_requests**：存储待处理请求的 HashMap
- **response_tx**：使用通道发送响应，避免阻塞

### 3. Hub 消息优先级

Hub 使用 event_matches!() 宏选择特定消息类型：

```rust
event_matches!(
    Message::DeviceControl { .. },
    Message::DeviceResponse { .. }
)
```

**优先级规则：**
1. DeviceControl > DeviceResponse（处理请求）
2. DeviceHeartbeat 最高优先级（保活连接）
3. ConfigUpdate 高优先级（配置热更新）

## 📊 请求-响应生命周期

### 请求创建

```
1. RPC Worker 收到 JSON-RPC 请求
2. 生成 correlation_id（原子递增计数器）
3. 创建 DeviceControl 消息
4. 发送到 Hub
5. 存储到 pending_requests HashMap（如果异步）
```

### 请求处理

```
1. Hub 投递 DeviceControl 到 DeviceManager
2. DeviceManager 查找目标 ModbusWorker
3. 通过 Hub 转发到 ModbusWorker
4. ModbusWorker 接收并加入操作队列
```

### 响应处理

```
1. ModbusWorker 完成操作
2. 发送 DeviceResponse 到 Hub
3. DeviceManager 接收 DeviceResponse
4. 通过 correlation_id 匹配请求
5. 通过 response_tx 发送响应给原始请求者
6. 移除 pending_requests 中的条目
```

### 超时处理

```
1. pending_requests 使用 ResponseSender with timeout()
2. 超时后自动移除对应条目
3. 记录超时警告日志
```

### 错误处理

```
1. 操作失败时在 DeviceResponse 中设置 error 字段
2. DeviceManager 将错误通过 response_tx 发送回请求者
3. 记录详细的错误日志
```

## 🎯 设计模式

### 1. 非阻塞响应
使用 mpsc::channel 避免 Worker 阻塞。

### 2. 背压控制
OperationQueue 限制最大并发操作数，防止系统过载。

### 3. 优雅降级
连接失败时使用指数退避策略，避免重连风暴。

### 4. 状态机
ModbusWorker 维护连接状态、心跳时间、操作统计。

## 🔧 配置管理

### worker_map 初始化
DeviceManager 从配置加载设备时创建 worker_map：

```rust
for device in &config.devices {
    worker_map.insert(device.id.clone(), format!("modbus_worker_{}", device.id));
}
```

### 配置热更新流程

```
1. ConfigLoader 检测到配置文件变化
2. 解析新配置（JSON 差异）
3. 通过 Hub 广播 ConfigUpdate 消息
4. ConfigUpdater 接收并应用更新
   - 更新设备列表（添加/移除）
   - 更新 device_states
   - 更新 worker_map
   - 必要时重启 ModbusWorker
```

## 📚 并发安全

### Hub 消息投递
Hub 使用非阻塞投递（try_send），高优先级消息不会被阻塞。

### Worker 消息处理
每个 Worker 的 run() 方法顺序处理消息：
1. 验证消息类型和来源
2. 执行相应操作
3. 发送响应
