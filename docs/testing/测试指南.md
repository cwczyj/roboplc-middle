# 测试指南

> 本指南说明如何测试 RoboPLC 系统，包括单元测试、集成测试和端到端测试。

## 🧪 测试结构

```
src/
├── api.rs                    # HTTP API 模块
├── workers/
│   ├── manager.rs           # 设备管理
│   ├── rpc_worker.rs         # JSON-RPC 服务
│   ├── modbus_worker.rs     # Modbus 控制
│   ├── config_updater.rs   # 配置热更新
│   └── http_worker.rs       # HTTP API 服务
└── tests/
    ├── e2e_tests.rs          # 端到端集成测试
    ├── functional_config_tests.rs  # 功能测试（配置）
    ├── functional_http_tests.rs   # 功能测试（HTTP）
    ├── functional_worker_tests.rs  # 功能测试（Worker）
    └── mock_modbus.rs        # Mock Modbus 设备
```

## 🏗️ 测试类型

### 1. 单元测试（Unit Tests）

**目的：** 测试单个函数和数据结构

**示例：**
```rust
#[test]
fn correlation_id_increments() {
    let id1 = TransactionId::new();
    let id2 = TransactionId::new();
    assert!(id2.id > id1.id);
}

#[test]
fn operation_queue_limits_concurrency() {
    let queue = OperationQueue::new(2);
    
    // 队列满
    for i in 0..3 {
        queue.push(QueuedOperation {
            operation: ModbusOp::ReadHolding { address: 100, count: 1 },
        });
    }
    
    assert!(!queue.can_start()); // 队列满，不能再添加
}
```

**文件位置：**
- 各模块的 `#[cfg(test)]` 模块
- tests/ 目录下的测试文件

### 2. 集成测试（Integration Tests）

**目的：** 测试多个组件协同工作

**示例：**
```rust
#[tokio::test]
async fn test_rpc_to_modbus_roundtrip() {
    // 设置测试环境
    let mut hub = roboplc::hub::Hub::new();
    
    // 启动 DeviceManager（模拟）
    let manager = hub.worker::<DeviceManager>("device_manager").unwrap();
    
    // 发送控制请求
    let correlation_id = 12345;
    let message = Message::DeviceControl {
        device_id: "test-device".to_string(),
        operation: Operation::SetRegister,
        params: json!({"address": "h100", "value": 42}),
        correlation_id,
    };
    
    hub.send(message);
    
    // 模拟响应（ModbusWorker）
    tokio::time::sleep(std::time::Duration::from_millis(100)).await;
    
    let response = hub.worker::<ModbusWorker>("modbus_worker_test-device").unwrap();
    
    // 验证响应
    tokio::time::sleep(std::time::Duration::from_millis(100)).await;
    
    let responses: response.collect_rx().await;
    assert_eq!(responses.len(), 1);
    
    let response = &responses[0];
    assert!(response.success);
}
```

**文件位置：**
- `tests/e2e_tests.rs` - 主要的集成测试文件

### 3. 功能测试（Functional Tests）

**目的：** 测试特定功能

#### 配置热更新测试
```rust
#[tokio::test]
async fn test_config_hot_reload() {
    // 测试配置热更新流程
    let mut hub = roboplc::hub::Hub::new();
    
    // 启动 DeviceManager
    let manager = hub.worker::<DeviceManager>("device_manager").unwrap();
    
    // 模拟配置更新
    let new_config = create_test_config();
    let config_json = serde_json::to_string(&new_config);
    
    let message = Message::ConfigUpdate { config: config_json };
    hub.send(message);
    
    // 验证更新
    tokio::time::sleep(std::time::Duration::from_millis(100)).await;
    
    // 检查 Config 变量是否更新
    let current_config = hub.variables().config().read().unwrap();
    assert_eq!(current_config.devices.len(), 2); // 原始 1 个，现在 2 个
}
```

#### 批量操作测试
```rust
#[tokio::test]
async fn test_batch_operations() {
    // 测试批量读写操作
    let mut hub = roboplc::hub::Hub::new();
    
    // 启动 DeviceManager 和 ModbusWorker
    let manager = hub.worker::<DeviceManager>("device_manager").unwrap();
    let worker = hub.worker::<ModbusWorker>("modbus_worker_batch").unwrap();
    
    // 发送批量写请求
    let correlation_id = 12346;
    let addresses = vec!["h100", "h101"];
    let values = vec![42, 43];
    let message = Message::DeviceControl {
        device_id: "test-device".to_string(),
        operation: Operation::WriteBatch,
        params: json!({
            "addresses": serde_json::to_string(&addresses),
            "values": serde_json::to_string(&values),
        }),
        correlation_id,
    };
    
    hub.send(message);
    
    // 验证响应
    tokio::time::sleep(std::time::Duration::from_millis(100)).await;
    
    let responses = manager.collect_responses(correlation_id).await;
    assert_eq!(responses.len(), 2);
}
```

**文件位置：**
- `tests/functional_config_tests.rs` - 配置相关功能测试
- `tests/functional_worker_tests.rs` - Worker 功能测试

### 4. Mock 测试

**目的：** 提供 Modbus 设备的 Mock 实现

**使用：**
- `tests/mock_modbus.rs` 文件提供 Mock 实现
- 用于不需要真实 Modbus 设备即可运行测试

**示例：**
```rust
#[cfg(test)]
mod mock_modbus {
    use super::*;
    
    pub fn create_mock_client() -> MockModbusClient {
        MockModbusClient {
            endpoint: "127.0.0.1:502".to_string(),
            connection: Some(Arc::new(MockConnection::new_connected())),
            unit_id: 1,
        }
    }
    
    pub fn create_mock_worker() -> ModbusWorker {
        // 创建 Mock ModbusWorker
    }
}
```

**文件位置：**
- `tests/mock_modbus.rs` - Mock 实现和测试工具

## 🚀 运行测试

### 运行所有测试
```bash
# 运行所有单元测试
cargo test

# 运行特定测试
cargo test test_name

# 带详细输出
cargo test -- -- --nocapture
```

### 运行集成测试
```bash
# 启动集成测试
cargo test --test e2e_tests

# 只运行 e2e_tests 中的测试
cargo test --test e2e_tests
```

### 调试模式
```bash
# 开发模式（模拟模式，使用配置文件）
ROBOPLC_SIMULATED=1 cargo test

# 生产模式
cargo test
```

## 📊 测试覆盖率

运行以下命令检查测试覆盖率：

```bash
cargo test --lib -- --no-fail-fast
```

## 🎯 最佳实践

### 1. 命序依赖测试
集成测试依赖：启动顺序正确（DeviceManager → ModbusWorker）

### 2. 隔离测试
每个测试独立运行，不互相干扰

### 3. 测试数据隔离
使用 portpicker 动态分配端口，避免端口冲突

### 4. 清理资源
测试结束后清理 Mock 资源和临时文件

### 5. 错误消息明确
测试中使用清晰的断言消息：
- `assert!()` - 断言失败时立即停止
- `assert_eq!()` - 验证预期值
- `assert_ne!()` - 验证不等于预期值
